# 网站需求、商业目标与建站路线

## 先判断六件事

1. 预期结果：展示、获客、销售、订阅、内容交付，还是社区互动。
2. 用户所在地区、使用语言和主要设备。
3. 是否需要登录、数据库、支付、订阅、上传或协作。
4. 内容更新频率，以及由谁负责更新。
5. 启动预算和每月可承受预算。
6. 是否有能力处理代码、故障和持续维护。

信息足够时直接判断；只有答案会改变建站路线时，才补问一个问题。

需求验证状态必须来自对话中的明确证据。未提供时，诊断只写“对话未提供验证证据”，不得据此断言用户处于需求验证前、验证中或其他阶段。

## 网站类型

| 类型 | 先验证 | 常见误区 |
| --- | --- | --- |
| 官网 | 访客能否理解业务，并完成联系或下单 | 只做视觉展示，没有明确行动入口 |
| 博客内容站 | 内容来源、发布频率和搜索需求 | 先建复杂系统，却没有持续内容 |
| 作品集 | 目标客户、代表案例和联系路径 | 只有图片，不说明能力、过程和结果 |
| 电商 | 商品、库存、物流、退款和目标地区支付 | 手工开发订单与支付核心 |
| SaaS 工具 | 核心任务、登录、数据、计费和支持 | 把 AI 生成的界面当作可直接上线的生产系统 |
| 社区 | 冷启动、治理、审核和留存 | 只有功能，没有种子用户和运营机制 |

## 建站方式

下表用于选择主要路线，不表示技术栈互斥；实际网站可以在 CMS 或成熟电商平台上组合定制应用组件。

| 方式 | 适合 | 不适合作为主要路线 |
| --- | --- | --- |
| AI 建站 / Vibe Coding | 原型、定制页面、轻量工具，且有人能维护 | 无人看护的关键交易系统 |
| SaaS 建站平台 | 官网、作品集、活动页和标准商业网站 | 高度定制的数据与权限逻辑 |
| 成熟电商平台 | 商品、订单、库存、税务和退款流程 | 单纯展示需求 |
| CMS | 内容量大、多编辑协作和持续发布 | 复杂交互工具 |
| 定制开发 | 需求稳定、预算明确且具备维护能力；或定制交互本身就是待验证假设的最小 MVP | 未验证且可用成熟方案验证的想法 |

定制 MVP 只实现验证该交互所需的最小范围。WordPress 只是 CMS 的一个选项，Vibe Coding 只是一条建站路线；不要按流行度决定。

## 上线风险门槛

AI / Vibe Coding 或新手搭建的网站，一旦涉及登录、私密数据、上传、API 密钥、破坏性操作、支付或受监管数据，应先按原型处理。生产上线前必须完成安全审查、核心旅程测试、备份与恢复验证、监控，并为身份、支付、存储等采用由服务商负责维护的服务。

## 推荐顺序

1. 先用一句话写清用户、问题和期望行动。
2. 选择能验证目标的最小网站类型与建站方式。
3. 先跑通一条核心路径，再完善品牌视觉和扩展功能。
4. 只深化当前阶段，不提前设计尚未验证的后续系统。

## 起步资源

- AI 原型可考察 v0.dev、lovable.dev、bolt.new；推荐前核验官方能力、代码导出和当前价格，再按维护条件最多选择两个。
- 视觉网站可比较 Framer 与 Webflow，以当前官方文档确认能力和限制。
- 内容站按需求比较 Ghost、WordPress 与现成内容网站发布服务，不默认指定 CMS。
- 电商优先核验目标市场的成熟平台，以及支付、退款和税务支持。

涉及具体产品时，遵循 [source-policy.md](source-policy.md)。

## 输出方式

给出当前判断、2-3 条理由、最多 3 个行动和 1-2 个风险；必要时补充一个备选。诊断区只描述已知事实、未知状态、现有成本或能力限制及其当前影响，不写“先按原型”“转向 MVP”等未来阶段或路线，不写“满足某条件时优先级会改变”等决策分支，也不用“最大风险是先建站”“风险是先开发/先买/先配置”等句式暗示禁令；阶段结论、路线、操作或禁令只放在唯一行动区。不得把“不会写代码”扩大成“无人维护代码”或其他未提供事实；预算与开发维护负担的经验判断必须标明“按常见实操经验”，且不编造精确报价。不要倾倒全部路线和资源段落。
